# EMOVEL Product Templates

## 1. Purpose

This document defines how EMOVEL structures digital products, templates, prompt packs, PDFs, shop assets, campaigns, bundles, and client-ready deliverables.

Its purpose is to help EMOVEL AI turn raw ideas into products that are clear enough to sell, useful enough to deliver, and structured enough to reuse.

An EMOVEL product should not feel like a random file. It should feel like a controlled asset with a defined buyer, purpose, format, value stack, and delivery logic.

The standard:

Every EMOVEL product must be understandable, usable, monetizable, and export-ready.

## 2. Product Philosophy

EMOVEL products are structured digital assets.

They are not content dumps. They are not aesthetic files with no commercial use. They are not vague resources labeled as products.

A strong EMOVEL product has:

- a specific audience
- a commercial problem
- a clear outcome
- a defined format
- a practical use case
- a premium presentation
- a delivery structure
- a path for reuse or implementation

Product philosophy:

- Build assets that solve a real execution problem.
- Package expertise into usable formats.
- Make the buyer's next step obvious.
- Keep the product focused.
- Avoid unnecessary complexity.
- Use structure as the premium signal.
- Make every product commercially legible.

The product should feel complete without feeling bloated.

## 3. What Counts As An EMOVEL-Ready Product

An EMOVEL-ready product is a digital asset that can be sold, delivered, used, and improved.

It must include:

- product name
- target audience
- problem solved
- transformation or outcome
- core mechanism
- included assets
- usage instructions
- delivery format
- pricing logic
- CTA or purchase path

Examples of EMOVEL-ready products:

- premium prompt pack
- offer architecture PDF
- landing page template
- campaign asset kit
- productized client deliverable
- Builder-generated page package
- template bundle
- commercial checklist
- Notion-style operating system
- exportable strategy document

Not EMOVEL-ready:

- a list of generic prompts
- a PDF with advice but no structure
- a landing page with no offer logic
- a template with no target buyer
- a campaign with no CTA
- a bundle of unrelated assets
- a product with no clear outcome

If the product cannot answer "who is this for and what does it help them do," it is not ready.

## 4. Digital Product Categories

EMOVEL can structure multiple product categories.

Primary categories:

- Prompt packs
- PDF guides
- Templates
- Shop assets
- Landing page systems
- Campaign kits
- Client deliverables
- Bundles
- Operating systems
- Commercial frameworks

Each category must have a clear use case.

Prompt packs help users produce controlled outputs.

PDFs help users understand, decide, or implement.

Templates help users start from a structured asset.

Shop assets help users buy a packaged solution.

Landing page products help users present and sell an offer.

Campaign products help users turn an offer into market-facing assets.

Bundles help users solve a larger workflow through connected products.

## 5. Prompt Pack Structure

A prompt pack is not a collection of random prompts.

It is a reusable prompt architecture for a specific commercial task.

Prompt pack structure:

1. Product name
2. Audience
3. Use case
4. Outcome
5. Prompt categories
6. Prompt instructions
7. Output format
8. Quality checks
9. Usage examples
10. Reuse guidance

Required sections:

- Introduction
- Who this is for
- What this helps you create
- How to use the prompts
- Prompt library
- Output standards
- Editing guidance
- Final checklist

Prompt categories may include:

- offer definition
- audience research
- product positioning
- landing page structure
- campaign angles
- social captions
- email launch copy
- objection handling
- export formatting

Quality rules:

- Every prompt must have a role.
- Every prompt must define context.
- Every prompt must specify output format.
- Every prompt must include constraints.
- Every prompt must produce a usable result.

Weak prompt pack:

- "50 AI prompts for entrepreneurs."

EMOVEL-ready prompt pack:

- "A structured prompt system for turning a product idea into offer positioning, landing page architecture, campaign angles, and launch copy."

## 6. PDF Product Structure

A PDF product should be a premium handoff document.

It should not be a long essay disguised as a product.

PDF product structure:

1. Cover / title page
2. Product promise
3. Audience definition
4. Problem framing
5. Core framework
6. Step-by-step implementation
7. Examples
8. Templates or worksheets
9. Quality checklist
10. Next steps

PDF rules:

- Use clear hierarchy.
- Keep sections modular.
- Make instructions practical.
- Avoid filler chapters.
- Include examples.
- Include implementation guidance.
- Make the document useful without explanation from the creator.

Good PDF products:

- Offer Architecture Guide
- Prompt System Builder
- Landing Page Structure Manual
- Campaign Planning Framework
- Premium Product Packaging Checklist

PDFs should be exportable, readable, and client-safe.

## 7. Template Product Structure

A template product gives the buyer a usable starting point.

It should reduce blank-page friction and preserve quality.

Template structure:

1. Template name
2. Intended use
3. Target user
4. Included sections
5. How to customize
6. Best use cases
7. What to avoid
8. Example filled version
9. Export or implementation instructions

Template types:

- landing page template
- offer template
- product description template
- campaign template
- email launch template
- client delivery template
- brand profile template
- prompt architecture template

Template rules:

- Do not make the buyer guess how to use it.
- Do not include sections without a purpose.
- Do not overload the template with options.
- Include guidance where judgment is needed.
- Keep the structure premium and commercial.

The template should feel like a working asset, not a decorative file.

## 8. Shop Asset Structure

A shop asset is a product listing that must sell clearly.

Shop asset structure:

1. Product title
2. Short positioning line
3. Product description
4. Who it is for
5. What is included
6. How to use it
7. Format and delivery
8. Value explanation
9. FAQ or objections
10. CTA

Shop listing rules:

- Make the product type clear.
- Use specific outcomes.
- Show what the buyer receives.
- Avoid hype.
- Avoid exaggerated earnings claims.
- Keep pricing aligned with product depth.
- Make delivery format obvious.

Good product title:

- "Offer Architecture Pack"

Weak product title:

- "Ultimate Business Success Bundle"

Shop assets should feel premium, focused, and immediately understandable.

## 9. Landing Page Product Structure

A landing page product is a structured commercial page system.

It must clarify the offer and move the reader toward action.

Landing page product structure:

1. Hero
2. Problem
3. Mechanism
4. Offer
5. Value stack
6. Proof logic
7. Objections
8. Pricing or package
9. CTA
10. FAQ

Landing page product rules:

- The hero must name the value.
- The problem must be specific.
- The mechanism must explain how the result is created.
- The offer must show concrete deliverables.
- The CTA must be direct.
- The page must avoid fake urgency.
- Every section must have a commercial role.

Landing page products can include:

- copy structure
- section framework
- CTA logic
- visual direction
- export package
- implementation notes

The page should sell through clarity, not pressure.

## 10. Campaign Product Structure

A campaign product turns an offer into market-facing assets.

Campaign structure:

1. Campaign goal
2. Audience
3. Offer angle
4. Core message
5. Content pillars
6. Asset list
7. Captions or copy
8. Visual direction
9. CTA system
10. Publishing guidance

Campaign assets may include:

- Instagram posts
- carousel copy
- email sequence
- launch captions
- ad angles
- product announcement copy
- story prompts
- CTA variations

Campaign rules:

- Start with the offer.
- Keep messaging consistent.
- Use specific commercial angles.
- Avoid random content ideas.
- Tie every asset to a campaign role.
- Do not use viral hooks that damage premium positioning.

Good campaign products create momentum without losing brand control.

## 11. Bundle Structure

A bundle is a connected product system.

It should not be a pile of unrelated files.

Bundle structure:

1. Bundle name
2. Core promise
3. Buyer profile
4. Products included
5. Recommended usage order
6. Workflow map
7. Value stack
8. Delivery formats
9. Implementation guidance
10. Upgrade or next step

Bundle rules:

- Every included product must support the same workflow.
- The order of use must be clear.
- The bundle should solve a larger problem than one product alone.
- Do not include weak assets to increase perceived size.
- Do not sell quantity over usefulness.

Good bundle:

- Offer + landing page + campaign kit for launching a premium digital product.

Weak bundle:

- 200 random templates, prompts, captions, and worksheets.

Premium bundles feel curated, not inflated.

## 12. Product Quality Checklist

Before approving a product, check:

- The audience is specific.
- The problem is clear.
- The outcome is believable.
- The product format is defined.
- The deliverables are concrete.
- The mechanism is visible.
- Usage instructions are included.
- The product can be sold with a clear description.
- The product can be delivered without explanation.
- The product avoids filler.
- The product avoids hype.
- The product avoids fake income claims.
- The product has pricing logic.
- The product has a clear CTA.
- The product feels premium.
- The product can be reused or implemented.

If the product fails on audience, outcome, or deliverable clarity, rebuild it before packaging.

## 13. Pricing And Packaging Rules

Pricing and packaging should reflect usefulness, depth, and buyer seriousness.

Pricing rules:

- Price for outcome and structure, not file count.
- Do not use fake discounts.
- Do not rely on inflated anchors.
- Do not make premium products feel cheap.
- Keep tiers easy to understand.
- Match price to buyer maturity and product depth.
- Explain value through commercial use.

Packaging rules:

- Name the product clearly.
- Make the buyer obvious.
- Show what is included.
- Show how to use it.
- State delivery format.
- Include a quality checklist or next step.
- Keep bonus assets limited and relevant.

Do not package confusion as abundance.

## 14. Delivery Format Rules

Delivery format must match the product's use case.

Common formats:

- PDF
- Markdown
- TXT
- Notion-style template
- prompt library
- landing page structure
- campaign document
- spreadsheet
- image assets
- ZIP package

Format rules:

- PDF for polished guides, frameworks, and client handoffs.
- Markdown for editable structured content.
- TXT for simple handoff and reuse.
- Templates for repeatable workflows.
- Prompt libraries for AI execution systems.
- Campaign docs for launch and marketing assets.

Every delivery should include:

- file name
- format
- intended use
- usage instructions
- version or date when useful

Exports should not expose internal metadata.

Delivery should feel clean and professional.

EMOVEL AI should use this file whenever it structures a product or productized deliverable.

- digital products

- landing pages
- campaigns
- bundles
- client deliverables
- export packages

EMOVEL AI should:

1. Identify the product category.
2. Define the buyer.

5. Choose the right structure.
6. Build the value stack.
7. Specify the delivery format.

- If the product feels bloated, remove weak assets.

- If the product relies on hype, replace it with practical value.

EMOVEL AI should not create products as random collections of content. It should create structured, premium, monetizable assets that can be used, sold, exported, and delivered.
